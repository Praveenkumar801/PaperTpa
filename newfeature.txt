# 📦 Overview

Develop a complete rework of `/tpa` featuring:

* GUI-driven teleport interaction
* Player reputation system
* Rating system
* Trap detection
* Anti-abuse safeguards
* Performance-optimized async database handling

The system must be modular, scalable, and cleanly structured.

---

# 🏗 Architecture Requirements (MANDATORY)

Use service-based architecture.

### Required Classes / Managers

* `TPARequestManager`
* `TeleportManager`
* `ReputationManager`
* `RatingManager`
* `GUIManager`
* `DataService` (SQLite access)
* `ConfigManager`

### Rules

* No business logic inside command classes
* No synchronous SQLite calls on main thread
* Use CompletableFuture or async scheduler for DB
* Cache player reputation data in memory
* Clean separation of concerns
* Proper listener-based event handling

---

# 🗄 Database Requirements (SQLite)

Use a single SQLite database file.

### Tables

**players**

* uuid (PK)
* total_sent
* total_received
* total_accepted
* total_denied
* trap_count
* rating_average
* rating_count

**ratings**

* id (PK)
* from_uuid
* to_uuid
* stars (1–5)
* trap_reported (boolean)
* timestamp

All queries must be async.

Cache player stats on join and update async.

# ⚙ Core Features

## 1️⃣ GUI-Based TPA System

When `/tpa <player>` is sent:

Target receives GUI with:

* ✅ Accept
* ❌ Deny
* 👤 View Player Info

No chat-only interaction unless GUI is disabled in config.

---

## 2️⃣ Player Info GUI Panel

Must display:

* Total TPA sent
* Total received
* Accepted
* Denied
* Trap %
* Average ⭐ rating
* Total ratings
* Reputation tier (Trusted / Neutral / Suspicious)

---

## 3️⃣ Teleport Flow

* Configurable warmup
* Configurable request expiry
* Cancel if:

  * Either player logs out
  * Player takes damage during warmup
  * Player enters combat (if detectable)
* Prevent teleport into unsafe locations (lava, void, inside block)

---

# ⭐ Rating System

After successful teleport:

* Wait configurable delay (default: 30s)
* Prompt teleported player with rating GUI
* Allow:

  * 1–5 stars
  * Optional trap report toggle

### Restrictions

* Only 1 rating per teleport instance
* Cannot rate if teleport duration < 5 seconds
* Rating only allowed if teleport completed

---

# 🚩 Trap Detection Logic

Trap is counted if:

* Player dies within configurable window (default 20s)
* OR player manually reports trap

Trap % formula:

```
(trap_count / total_accepted) * 100
```

If trap % ≥ 25% → Reputation tier becomes Suspicious.

---

# 🏆 Reputation Tiers

* 🟢 Trusted → 4.5⭐+
* 🟡 Neutral → 3.0–4.4⭐
* 🔴 Suspicious → Trap % ≥ 25%

Tier must be calculated dynamically.
